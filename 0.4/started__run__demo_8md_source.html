1 # Run Demos #
2 
3 For simplicity and user's convenience, there are only 2 basic application demos which are *Device* and *Host* respectively. Each application demo, however, has a few projects, each for its supported RTOS. For instance in the /demo/device you will find:
4 
5 - device\_os\_none for no RTOS
6 - device\_freertos for freeRTOS
7 - device\_cmsis_rtx for ARM CMSIS with RTX implemenation
8 
9 To be able to have the same application code running across RTOSes, the application make use of the "internal" **OSAL layer**. Thus this makes the application code a bit weird and over-complicated than it should be in some (many) cases. This is absolutely not necessary in product development. User can just use the native API function of supported RTOS or a state machine or blocking wait in case of none OS. For example, instead of the blinking task in application
10 
11 ~~~{.c}
12 OSAL_TASK_FUNCTION( led_blinking_task , p_task_para)
13 {
14  OSAL_TASK_LOOP_BEGIN
15 
16  static uint32_t led_on_mask = 0;
17 
18  osal_task_delay(led_blink_interval_ms);
19 
20  board_leds(led_on_mask, 1 - led_on_mask);
21  led_on_mask = 1 - led_on_mask; // toggle
22 
23  OSAL_TASK_LOOP_END
24 }
25 ~~~
26 
27 can be written in FreeRTOS's native API
28 
29 ~~~{.c}
30 void led_blinking_task( void * p_task_para )
31 {
32  while(1)
33  {
34  static uint32_t led_on_mask = 0;
35 
36  // FreeRTOS API's vTaskDelay is used in place of osal_task_delay. Note it takes input parameter in tick
37  vTaskDelay( (led_blink_interval_ms * TUSB_CFG_TICKS_HZ) / 1000);
38 
39  board_leds(led_on_mask, 1 - led_on_mask);
40  led_on_mask = 1 - led_on_mask; // toggle
41  }
42 }
43 ~~~
