12 ## Download
13 
14 tinyusb uses github as online repository https://github.com/hathach/tinyusb since it is the best place for open source project.
15 
16 If you are using Linux, you already know how to what to do. But If Windows is your OS, I would suggest to install [git](http://git-scm.com/) and front-end gui such as [tortoisegit](http://code.google.com/p/tortoisegit) to begin with.
17 
18 After downloading/cloning, the code base is composed of
19 
20 Folder | Description
21 ----- | -------------
22 boards | Source files of supported boards
23 demos | Source & project files for demonstration application
24 mcu | Low level mcu core & peripheral drivers (e.g CMSIS )
25 tests | Unit tests for the stack
26 tinyusb | All sources files for tinyusb stack itself.
27 vendor | Source files from 3rd party such as freeRTOS, fatfs etc ...
28 
29 *demos* is the folder where all the application & project files are located. There are demos for both device and hosts. For each, there are different projects for each of supported RTOS.

31 ## Add tinyusb to your project
32 
33 It is relatively simple to incorporate tinyusb to your (existing) project
34 
35 1. Copy core folder **tinyusb** to your project. Let's say it is *your_project/tinyusb*
36 2. Add all the .c in the core folder to your project settings (uvproj, ewp, makefile)
37 3. Add *your_project/tinysb* to your include path. Also make sure your current include path also contains the configuration file tusb_config.h. Or you could simply put the tusb_config.h into the tinyusb folder as well.
38 4. Make sure all required macros are all defined properly in tusb_config.h (configure file in demo application is sufficient, but you need to add a few more such as TUSB_CFG_MCU, TUSB_CFG_OS, TUSB_CFG_OS_TASK_PRIO since they are passed by IDE/compiler to maintain a unique configure for all demo projects).
39 5. If you use the device stack, make sure you have created/modified usb descriptors for your own need. Ultimately you need to fill out required pointers in tusbd_descriptor_pointers for that stack to work.
40 6. Add tusb_init() call to your reset initialization code.
41 7. Implement all enabled classes's callbacks.
42 8. If you dont use any RTOSes at all, you need to continuously and/or periodically call tusb_task_runner() function. Most of the callbacks and functionality are handled and invoke within the call of that task runner.
43 
44 ~~~{.c}
45 int main(void)
46 {
47  your_init_code();
48  tusb_init(); // initialize tinyusb stack
49 
50  while(1) // the mainloop
51  {
52  your_application_code();
53 
54  tusb_task_runner(); // handle tinyusb event, task etc ...
55  }
56 }
57 ~~~
