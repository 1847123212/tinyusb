1 # Coding Standards #
2 
3 C is a dangerous language by itself, plus tinyusb make use of goodies features of C99, which saves a tons of code lines (also means save a tons of bugs). However, those features can be misused and pave the way for bugs sneaking into. Therefore, to minimize bugs, the author try to comply with published Coding Standards like:
4 
5 - [MISRA-C](http://www.misra-c.com/Activities/MISRAC/tabid/160/Default.aspx)
6 - [Power of 10](http://spinroot.com/p10/)
7 - [Jet Propulsion Laboratory (JPL) for C](http://lars-lab.jpl.nasa.gov)
8 
9 Where is possible, standards are followed but it is almost impossible to follow all of these without making some exceptions. I am pretty sure this code base violates more than what are described below, if you can find any, please report it to me or file an issue on github.
10 
11 ## MISRA-C 2004 Exceptions ##
12 
13 MISRA-C is well respected & a bar for industrial coding standard.
14 
15 - **Rule 2.2: use only**
16 
17  It has long passed the day that C99 comment style // will cause any issues, especially compiler's C99 mode is required to build tinyusb.
18 
19 - **Rule 8.5: No definitions of objects or function in a header file**
20 
21  function definitions in header files are used to allow 'inlining'
22 
23 - **Rule 14.7: A function shall have a single point of exit at the end of the function**
24 
25  Unfortunately, following this rule will have a lot of nesting if-else, I prefer to exit as soon as possible with assert style and flatten if-else.
26 
27 - **Rule 18.4: Unions shall not be used**
28 
29  sorry MISRA, union is required to effectively mapped to MCU's registers
30 
31 - expect to have more & more exceptions.
32 
33 ## Power of 10 ##
34 
35 is a small & easy to remember but yet powerful coding guideline. Most (if not all) of the rules here are included in JPL. Because it is very small, all the rules will be listed here, those with *italic* are compliant, **bold** are violated.
36 
37 1. *Restrict to simple control flow constructs*
38 
39  yes, I hate goto statement, therefore there is none of those here
40 
41 2. *Give all loops a fixed upper-bound*
42 
43  one of my favorite rule
44 
45 3. *Do not use dynamic memory allocation after initialization*
46 
47  the tinyusb uses the static memory for all of its data.
48 
49 4. **Limit functions to no more than 60 lines of text**
50 
51  60 is a little bit too strict, I will update the relaxing number later
52 
53 5. *Use minimally two assertions per function on average*
54 
55  not sure the exact number, but I use a tons of those assert
56 
57 6. *Declare data objects at the smallest possible level of scope*
58 
59  one of the best & easiest rule to follow
60 
61 7. *Check the return value of non-void functions, and check the validity of function parameters*
62 
63  I did check all of the public application API's parameters. For internal API, calling function needs to trust their caller to reduce duplicated check.
64 
65 8. **Limit the use of the preprocessor to file inclusion and simple macros**
66 
67  Although I prefer inline function, however C macros are far powerful than that. I simply cannot hold myself to use, for example X-Macro technique to simplify code.
68 
69 9. *Limit the use of pointers. Use no more than two levels of dereferencing per expression*
70 
71  never intend to get in trouble with complex pointer dereferencing.
72 
73 10. *Compile with all warnings enabled, and use one or more source code analyzers*
74 
75  I try to use all the defensive options of gnu, let me know if I miss some.
76  >-Wextra -Wswitch-default -Wunsafe-loop-optimizations -Wcast-align -Wlogical-op -Wpacked-bitfield-compat -Wnested-externs -Wredundant-decls -Winline
